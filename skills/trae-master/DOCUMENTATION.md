## 1. 框架概述

### 什么是 Trae Master？

这是一个**模块化的技能开发和执行框架**，可以把它想象成技能的"操作系统"。

```
┌─────────────────────────────────────────┐
│              Trae Master               │
│            Trae Master 框架             │
│                                         │
│  你的需求 ──► 框架调度 ──► 技能执行     │
│              │                          │
│              ▼                          │
│     自动总结记录（汇报完+用户确认）      │
└─────────────────────────────────────────┘
```

### 核心特点

| 特点 | 说明 |
|------|------|
| 模块化 | 每个技能独立自包含 |
| 可复用 | 技能可在不同场景复用 |
| 标准化 | 统一接口规范 |
| 触发记录 | 用户确认后才记录上下文 |

## 4. 各层详解

### 4.1 core/ 核心层

**框架自带的基础能力，所有技能都依赖它。**

```
core/
├── PROTOCOL.md            ← 通信协议/阶段切换规则
├── dispatch-rules.md      ← 调度与通信规则
├── acceptance-criteria.md ← 交付验收标准
├── context-log/           ← 上下文自动总结记录（汇报完+用户确认，失败/终止例外；只记有用信息）
└── meta-skill/            ← 技能调度器（调度其他技能）
```

#### context-log/ 上下文记录系统

| 文件/目录 | 功能 |
|-----------|------|
| SKILL.md | 记录规则、格式、触发条件 |
| logs/ | 每日日志目录 |
| logs/YYYY-MM-DD.md | 每天一个日志文件 |

**记录触发条件：用户说"可以"后才记录**

---

#### meta-skill/ 技能调度器

| 文件 | 功能 |
|------|------|
| SKILL.md | 调度规则、如何创建新技能 |
| REGISTRY.md | 技能注册表（有哪些技能） |

**作用：识别用户意图，调度到正确的技能**

---

## 5. 使用示例

### 示例：用户输入一个需求

**用户说：**
> "我需要一个用户登录功能"

**AI 做的：**

```
1. 执行 FRAMEWORK.md 的启动前校验（Pre-flight Check），确保已读取所有框架文件
2. 识别这是"工作流需求"
3. 调度 skills/workflow-engine
4. 按阶段串行执行（PM → RD → QA）（注：阶段内部专员为动态按需调度）：
   - PM阶段：PM组长分析需求，按需分配专员
   - RD阶段：RD组长分析需求，按需分配专员
   - QA阶段：QA组长分析需求，按需分配专员
5. 每个阶段展示给用户
6. 用户确认后记录到 context-log，并校验日志已记录后才进入下一阶段
```

**core/context-log/logs/2026-04-19.md 记录内容：**

```markdown
## 阶段记录

### [14:30:00] REQ-2026-001 - PM阶段完成
- **用户确认**: 是
- **PM产出**: PRD文档、需求分析
- **Boss反馈**: 通过

---

## 完整需求总结（在最下面）

### REQ-2026-001: 用户登录功能

- **完成时间**: 2026-04-19 18:00:00
- **状态**: ✅ 完成

#### PM产出
- 需求文档已完成
- 用户画像已分析

#### RD产出
- 技术方案已设计
- 数据库设计已完成

#### QA产出
- 测试用例已设计
- 功能测试通过

#### 用户最终确认
- **确认时间**: 2026-04-19 18:00:00
- **用户说**: "可以"
```
