# Trae Master 框架使用指南

## 一、框架是什么

Trae Master 是一个 AI 驱动的研发工作流框架，模拟真实团队协作模式：Boss（当前AI）→ 组长 → 专员，按 PM → RD → QA 三阶段严格顺序执行需求。

核心理念：**AI 不是一个人干所有事，而是像一个真实团队一样分工协作。**

## 三、角色体系

| 角色 | 由谁担任 | 职责 | 通信范围 |
|------|---------|------|---------|
| Boss | 当前使用的AI | 接收需求、审核产出、阶段切换、数据请求中转 | 用户 + 3个组长 |
| PM Leader | AI调度 | 需求分析、动态调度PM专员 | Boss + PM组内专员 |
| RD Leader | AI调度 | 技术设计、代码优先查询、动态调度RD专员 | Boss + RD组内专员 |
| QA Leader | AI调度 | 测试验证、动态调度QA专员 | Boss + QA组内专员 |
| 专员 | AI调度 | 执行专项任务（需求理解/技术设计/功能测试等） | 同组专员 |

**关键规则**：
- Boss 是唯一枢纽，组长之间不能直接通信
- 专员只能与同组专员通信
- 所有跨组通信必须通过 Boss 中转

## 四、工作流程

### 完整流程

```
用户提交需求
    ↓
Boss 接收并解读需求
    ↓
Boss 分配给 PM Leader
    ↓
PM Leader 动态调度 PM 专员（需求理解→数据分析→文档编写→需求审查）
    ↓
PM Leader 汇总提交给 Boss
    ↓
Boss 审核 → 向用户汇报 → 用户确认"可以" → 记录日志 → 校验日志
    ↓
Boss 分配给 RD Leader
    ↓
RD Leader Round 1: 代码查询（项目结构/接口/IDL）← 必须先查代码！
RD Leader Round 2: 技术方案设计（基于代码查询结果）
RD Leader Round 3: 编码实现 + 质量保障
    ↓
RD Leader 汇总提交给 Boss
    ↓
Boss 审核 → 向用户汇报 → 用户确认"可以" → 记录日志 → 校验日志
    ↓
Boss 分配给 QA Leader
    ↓
QA Leader 动态调度 QA 专员（测试用例→功能测试→资金安全测试→测试交付）
    ↓
QA Leader 汇总提交给 Boss
    ↓
Boss 审核 → 向用户汇报 → 用户确认"可以" → 记录日志 → 校验日志
    ↓
🎉 流程完成
```

### 阶段切换前置条件

进入下一阶段**必须同时满足**以下4个条件：

1. 上一阶段 Boss 审核通过
2. 上一阶段用户确认"可以"
3. 上一阶段日志已记录
4. 上一阶段日志已校验

**缺一不可，否则禁止进入下一阶段。**

## 六、核心规则

### 1. 数据真实性（最重要）

**严禁推测、判断、推断、预测任何虚假信息，所有数据必须真实可验证。**

| 场景 | 正确行为 | 错误行为 |
|------|---------|---------|
| 数据缺失 | 立即上报，等待真实数据 | 自行推测或编造数据 |
| 信息不确定 | 明确标注 DATA_MISSING | 凭印象或推断给出结论 |
| 需要额外信息 | 通过层级上报请求用户提供 | 假设或猜测缺失信息 |

### 2. 动态调度

组长根据需求内容决定调度哪些专员，不需要的专员不调度。只有存在依赖关系时才需要等待前置专员完成，无依赖关系的专员可并行执行。

### 3. RD 代码优先

RD 阶段 Round 1 **必须先查代码**（项目结构/接口定义/现有实现/IDL），再设计方案。严禁在未查代码的情况下设计架构方案。若违反，Boss 审核必须打回。

### 4. 用户确认触发记录

只有在用户明确确认"可以"后，才会记录日志到 context-log。Boss 审核通过不等于记录，必须等用户确认。

## 七、安装与使用

### 前置条件

- 安装了 Trae IDE（或任何支持 Skills 的 AI IDE）
- 将 `skills/` 目录放置在 `.trae/skills/` 下

### 目录结构要求

```
你的项目/.trae/skills/
├── trae-master/     ← 框架本体
├── boss/            ← Boss智能体
├── pm-leader/       ← PM组长
├── rd-leader/       ← RD组长
└── qa-leader/       ← QA组长
```

### 使用步骤

1. **启动框架**：在对话中提及需要执行研发工作流，AI 会读取框架文件
2. **完成启动前校验**：AI 会按照 FRAMEWORK.md 中的 Pre-flight Check 读取所有框架文件
3. **提交需求**：向 AI 描述你的需求，AI 以 Boss 身份接收
4. **跟进流程**：AI 会按 PM → RD → QA 顺序执行，每个阶段结束会向你汇报
5. **确认或打回**：每个阶段汇报后，你确认"可以"才会进入下一阶段
6. **流程完成**：三个阶段全部通过后，AI 输出完成报告

### 自定义知识库

框架默认不包含业务域知识库。你可以创建自己的知识库：

1. 在 `trae-master/knowledge/` 下创建新目录，如 `my-domain/`
2. 创建 `SKILL.md` 文件，添加 frontmatter：
   ```yaml
   ---
   name: "my-domain"
   description: "Your domain knowledge description"
   ---
   ```
3. 在 `core/meta-skill/REGISTRY.md` 中注册，状态设为 Active
4. 在 `FRAMEWORK.md` 的 Pre-flight Check 中添加读取项

## 十一、知识库详细配置

### 知识库是什么

知识库是框架的"业务大脑"，让 AI 了解你的业务领域。没有知识库，AI 只能做通用分析；有了知识库，AI 能理解你的业务术语、链路、架构，产出更贴合实际的结果。

### 知识库目录结构

```
trae-master/knowledge/
├── bnpl-domain/              ← 示例：BNPL业务域知识库（需用户自行创建）
│   └── SKILL.md              ← 知识库入口文件
├── methods/                  ← 方法库（框架自带）
│   ├── SKILL.md
│   ├── README.md
│   ├── karpathy-guidelines-examples.md
│   └── bilingual-conversion.md
└── templates/                ← 模板库（框架自带）
    ├── SKILL.md
    ├── README.md
    ├── code-review.md
    ├── meeting-notes.md
    ├── project-plan.md
    ├── status-report.md
    └── test-plan.md
```

### 创建业务域知识库（完整步骤）

#### 第1步：创建目录和 SKILL.md

在 `trae-master/knowledge/` 下创建你的业务域目录：

```
trae-master/knowledge/my-domain/
└── SKILL.md
```

SKILL.md 必须包含 frontmatter 和业务知识内容：

```markdown
---
name: "my-domain"
description: "My business domain knowledge base"
---

# My Domain Knowledge Base | 我的业务领域知识库

## 1. Business Overview | 业务概览
（描述你的业务是什么、做什么、服务谁）

## 2. Core Business Flow | 核心业务链路
（描述你的核心业务流程，如：用户注册→下单→支付→发货）

## 3. Key Terminology | 关键术语
（列出你的业务术语，让 AI 正确理解而不是泛化）

## 4. System Architecture | 系统架构
（描述你的技术架构、服务划分、核心模块）

## 5. Data Model | 数据模型
（描述核心数据实体和关系）

## 6. API Overview | 接口概览
（列出核心 API 和调用关系）

## 7. Business Rules | 业务规则
（列出关键的业务约束和规则）
```

#### 第2步：编写知识库内容

**推荐的知识库章节结构**（参考 bnpl-domain）：n
| 章节 | 内容 | 重要性 |
|------|------|--------|
| 业务概览 | 业务是什么、服务谁、覆盖哪些市场 | 必须 |
| 核心业务链路 | 用户从进入到完成的关键路径 | 必须 |
| 关键术语 | 业务专有名词和含义 | 必须 |
| 系统架构 | 技术架构、服务划分 | 建议 |
| 数据模型 | 核心实体和关系 | 建议 |
| 接口概览 | 核心 API | 按需 |
| 业务规则 | 关键约束和规则 | 必须 |

**编写原则**：
- 使用**中英双语**格式（`## Title | 标题`），AI 在中英文场景下都能准确理解
- 内容必须**真实准确**，不要写推测或假设的内容
- 术语要**明确**，避免 AI 泛化理解
- 链路要**完整**，从起点到终点不要有断点

#### 第3步：注册到 REGISTRY.md

打开 `trae-master/core/meta-skill/REGISTRY.md`，在 Knowledge Base 部分添加：

```markdown
| my-domain | knowledge/my-domain | Active | My business domain knowledge description |
```

#### 第4步：添加到 Pre-flight Check

打开 `trae-master/FRAMEWORK.md`，在 Pre-flight Check 清单中添加：

```markdown
- [ ] 已读取 knowledge/my-domain/SKILL.md
```

同时更新 SKILL.md 和 FRAMEWORK.md 中的文件数量（+1）。

#### 第5步：更新目录结构图

在 `FRAMEWORK.md` 和 `DOCUMENTATION.md` 的目录结构图中添加你的知识库目录。

## 十二、个人信息配置

### 个人信息是什么

个人信息是让 AI 了解"你是谁"的关键上下文，包括你的公司、团队、业务方向、技术栈、开发环境等。有了这些信息，AI 的分析会更贴合你的实际工作场景。

### 个人信息放在哪里

个人信息需要写入**两个文件**（两个文件都需要写入，确保 Boss 和 PM 专员都能获取）：

| 文件 | 路径 | 作用 |
|------|------|------|
| Boss 指令 | `skills/workflow-engine/boss/instructions.md` | Boss 接收需求时理解你的业务背景 |
| 需求理解专员 | `skills/workflow-engine/pm/specialists/01-requirement-understanding/instructions.md` | PM 分析需求时理解你的业务视角 |

### 个人信息格式

在上述两个文件的末尾（`---` 分隔线之后），添加以下内容：

```markdown
---

## Information Synchronization & User Background

以下内容来自用户的信息同步，请作为你的长期记忆与上下文：
```

### 配置步骤

#### 第1步：准备你的个人信息

按照上面的模板，填写你的真实信息。**不要编造或推测**，只写你确定的内容。

#### 第2步：写入 boss/instructions.md

1. 打开 `skills/workflow-engine/boss/instructions.md`
2. 在文件末尾添加 `---` 分隔线
3. 在分隔线后粘贴你填好的个人信息

#### 第3步：写入 01-requirement-understanding/instructions.md

1. 打开 `skills/workflow-engine/pm/specialists/01-requirement-understanding/instructions.md`
2. 在文件末尾添加 `---` 分隔线
3. 在分隔线后粘贴同样的个人信息

#### 第4步：验证

启动框架后，向 AI 提交一个需求，观察 AI 是否正确理解了你的业务背景。如果 AI 的分析偏离了你的业务方向，检查个人信息是否完整、术语是否明确。

### 个人信息编写原则

| 原则 | 说明 |
|------|------|
| 真实准确 | 只写你确定的信息，不确定的不要写 |
| 具体明确 | "我做贷前"比"我做金融"好，"Go + Kitex"比"后端"好 |
| 术语标注 | 你的团队特有的术语一定要标注含义，避免 AI 泛化理解 |
| 业务链路 | 写清楚核心业务从起点到终点的完整路径 |
| 分析偏好 | 告诉 AI 你希望从什么视角分析问题 |
| 定期更新 | 当你的业务方向、技术栈变化时，及时更新个人信息 |

## 十三、常见问题八、审核与打回机制

### Boss 审核流程

每个阶段完成后：

1. 组长汇总所有专员产出，提交给 Boss
2. Boss 执行数据真实性校验：
   - 所有专员产出的数据是否有明确来源？
   - 是否存在未标注的推断/推测/预测内容？
   - 缺失数据是否已标注 DATA_MISSING？
   - 关键数据是否与原文一致？
3. Boss 通过 → 向用户汇报 → 用户确认 → 记录日志 → 下一阶段
4. Boss 不通过 → 附带错误包打回组长 → 组长重新调度相关专员

### 3次拒绝机制

同一阶段 Boss 最多拒绝3次。第3次拒绝后，Boss 向用户提供选项：

| 选项 | 说明 |
|------|------|
| 重试(Retry) | 从头重启本阶段 |
| 跳过(Skip) | 跳过本需求，继续下一个 |
| 简化(Simplify) | 缩减范围以适应限制 |
| 终止(Abort) | 停止工作流，记录失败 |

## 九、数据缺失处理

当专员报告数据缺失时，按以下链路处理：

```
专员报告 DATA_MISSING
    ↓
组长汇总所有数据缺失请求
    ↓
组长在汇总输出中列出所有缺失数据
    ↓
Boss 汇总后向用户请求数据
    ↓
用户提供数据
    ↓
Boss 转发给组长
    ↓
组长继续执行
```

**严禁任何角色自行编造或推测缺失数据。**

## 十、需求记录

每个需求会在 `skills/workflow-engine/requirements/active/REQ-YYYY-NNN/` 下创建目录，包含：

| 文件 | 说明 |
|------|------|
| metadata.md | 需求基本信息（ID、标题、优先级等） |

| pm-record.md | PM阶段记录 |

## 十一、常见问题

**Q: 可以跳过某个阶段吗？**
A: 不可以。PM → RD → QA 是严格顺序，每个阶段都是下一阶段的输入。

**Q: 可以只调用某个组长吗？**
A: 可以。pm-leader、rd-leader、qa-leader 可以独立调用，但此时没有 Boss 审核，产出质量由你自行把控。

**Q: 专员是固定的还是动态的？**
A: 动态的。组长根据需求内容决定调度哪些专员，不需要的专员不调度。

**Q: RD 阶段为什么必须先查代码？**
A: 避免脱离项目实际状态凭空设计方案。先查代码了解现有实现，再基于真实代码设计方案，确保方案可落地。

**Q: 数据缺失时怎么办？**
A: 立即标注 DATA_MISSING 并上报，等待用户提供真实数据。绝不允许自行推测或编造。

**Q: 如何添加自己的业务知识库？**
A: 参见"自定义知识库"章节，在 knowledge/ 下创建目录并注册到 REGISTRY.md。
